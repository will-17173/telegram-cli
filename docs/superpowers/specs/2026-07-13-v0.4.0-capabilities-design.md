# Telegram CLI v0.4.0 Capabilities Design

**Date:** 2026-07-13
**Status:** Approved
**Target release:** 0.4.0

## Summary

Version 0.4.0 adds the remaining day-to-day Telegram client capabilities identified during comparison with `skillhq/telegram`: unread inbox summaries, direct online reads and searches, contact lookup, notification muting, chat folders, administrator-group discovery, logout with retained local data, working ownership transfer with 2FA, Markdown output, incremental Markdown archives, and a global remote-write switch.

The release keeps the existing command names and local-first behavior stable. In particular, `search` remains a local SQLite query, `history` and `sync` remain the commands that persist Telegram messages into SQLite, and all existing commands keep their current default behavior.

All features in this document ship together as v0.4.0. Implementation is divided into internal batches so each domain remains independently testable.

## Goals

- Add all capabilities agreed for v0.4.0 without changing existing command semantics.
- Keep Telegram-specific behavior behind focused adapters.
- Prefer mtcute high-level APIs and use raw TL only for notification settings.
- Give every finite command human, JSON, YAML, and Markdown output.
- Preserve multi-account isolation and existing local SQLite data across logout/login.
- Make remote writes centrally disableable without disabling reads or local operations.
- Make multi-chat Markdown archives incremental, retryable, and safe after interrupted writes.
- Preserve structured output schema version 1 and existing non-TTY defaults.

## Non-goals

- Compatibility aliases for `skillhq/telegram` command names.
- Changing `search` from local to remote search.
- Persisting messages fetched by `read` or `search-online` into SQLite.
- Contact creation, deletion, editing, or notes.
- Folder creation, deletion, renaming, or reordering.
- A continuously running archive daemon.
- Automatic reconciliation of edited or deleted Telegram messages during incremental archive runs.
- Supplying the Telegram 2FA password through arguments, environment variables, files, or structured input.
- Enabling Markdown output for the long-running `listen` stream.

## Architecture

### Layers

The existing layering remains in force:

- `src/cli/` registers global behavior and renders process-level output.
- `src/commands/` defines Commander contracts and performs preflight validation.
- `src/services/` coordinates workflows without printing.
- `src/telegram/` isolates mtcute and raw TL types.
- `src/account/` owns account state and per-account paths.
- `src/storage/` continues to own the local message database.
- `src/presenters/` owns human, JSON, YAML, Markdown, and Ink rendering.

New Telegram capabilities are split by domain instead of expanding the existing `TelegramClientAdapter` into a monolith:

- `TelegramDialogAdapter`: dialogs, online history, online search, and group discovery.
- `TelegramContactAdapter`: contact list and user resolution.
- `TelegramNotificationAdapter`: peer notification state and mute changes.
- `TelegramFolderAdapter`: folder lookup, membership inspection, and chat inclusion/exclusion.
- `TelegramAccountSessionAdapter`: remote logout and authenticated identity.

The concrete mtcute client composes these adapters. Tests use corresponding fakes. Group management remains in the existing `TelegramGroupManagementAdapter`.

### Services

- `DialogService`: inbox, direct read, online search, and administrator-group filtering.
- `ContactService`: contact list and contact details.
- `NotificationService`: notification info, mute, and unmute.
- `FolderService`: folder list/info and chat add/remove.
- `ArchiveService`: range selection, history pagination, Markdown generation, manifest handling, and optional media downloads.
- `AccountSessionService`: logout/login state transitions.
- `WriteAccessPolicy`: one policy used by all Telegram remote-write paths.

Commands remain thin: parse options, validate conflicts, select an account context, call one service, and render its `HandlerResult`.

## Command Contracts

### Shared options

Every finite command supports account selection and four output modes:

```text
--account <name>
--json | --yaml | --markdown
```

The three explicit format flags are mutually exclusive. `OUTPUT=markdown` becomes a valid default alongside the existing output environment values. Explicit command flags continue to override `OUTPUT`. Without an explicit format, TTY and non-TTY behavior remains unchanged. `listen` remains a streaming command and does not accept `--markdown`.

Positive integer limits are validated before a Telegram client is created.

### Inbox

```text
tg inbox [--limit <chats>]
```

- Default limit: 100 unread chats.
- Valid limit: 1 through 500.
- Iterates dialogs until the requested number of unread dialogs is found or dialogs are exhausted.
- Returns total unread messages across returned dialogs, the returned unread-chat count, each dialog's unread count, unread mentions/reactions where available, mute state, and last-message summary.
- Does not fetch every unread message and does not mark anything read.

### Direct online read

```text
tg read <chat>
  [-n, --limit <messages>]
  [--since <relative-or-iso>]
  [--until <relative-or-iso>]
```

- Default limit: 50 messages.
- Valid limit: 1 through 1000.
- Reads Telegram directly using mtcute history pagination.
- Does not insert messages into SQLite.
- Returns newest-first output after applying the requested time window.

### Online search

```text
tg search-online <query>
  [--chat <chat>]
  [-n, --limit <messages>]
  [--since <relative-or-iso>]
  [--until <relative-or-iso>]
```

- Default limit: 50 messages.
- Valid limit: 1 through 1000.
- With `--chat`, uses chat-scoped Telegram search.
- Without `--chat`, uses Telegram global search.
- Does not insert results into SQLite.
- The existing `search` command remains local-only and unchanged.
- Telegram's own global-search result limits are surfaced in help/documentation rather than treated as a CLI bug.

### Contacts

```text
tg contact list [--limit <contacts>]
tg contact info <user-or-phone>
```

- `contact list` defaults to 100 results and accepts 1 through 1000.
- `contact info` accepts a username, marked/numeric user ID, or phone number.
- Username/ID lookup uses mtcute user resolution.
- Phone lookup uses `resolvePhoneNumber()` followed by user lookup.
- Telegram privacy and relationship restrictions remain authoritative; an unresolvable phone number returns `phone_not_resolvable`.
- v0.4.0 does not modify the user's Telegram contact list.

### Notification settings

```text
tg notification info <chat>
tg notification mute <chat> [duration]
tg notification unmute <chat>
```

- Duration accepts a positive integer followed by `s`, `m`, `h`, `d`, or `w`, plus `forever`.
- Omitting duration from `mute` means `forever`.
- `forever` maps to Telegram's maximum accepted mute timestamp.
- `info` reports explicit mute state, effective mute-until time, and whether the dialog currently appears muted.
- `mute` and `unmute` update only `muteUntil`, leaving unrelated notification settings unspecified.
- Implementation uses typed raw TL calls to `account.getNotifySettings` and `account.updateNotifySettings` because mtcute 0.30.3 has no high-level mutation method.

### Chat folders

```text
tg folder list
tg folder info <folder-id-or-title>
tg folder chat add <folder-id-or-title> <chat>
tg folder chat remove <folder-id-or-title> <chat>
```

- Folder ID is the stable automation identifier.
- A title resolving to multiple folders returns `ambiguous_folder` and lists matching IDs.
- `folder info` includes folder rules and the dialogs currently matching the folder.
- Adding an already included chat succeeds with `changed: false`.
- Removing a chat not currently included succeeds with `changed: false`.
- Add resolves the chat peer, removes it from `excludePeers`, and adds it to `includePeers` when needed.
- Remove deletes the peer from `includePeers` and `pinnedPeers`. If category rules would still include the chat, remove adds it to `excludePeers`.
- Imported/shared chat-list folders that cannot express exclusions return a capability error rather than pretending removal succeeded.

### Group discovery

```text
tg group list [--admin] [--limit <chats>]
```

- Default limit: 100 results.
- Valid limit: 1 through 500.
- Includes basic groups, supergroups, and broadcast channels.
- `--admin` keeps peers where the current user is administrator or creator.
- Uses dialog peer metadata (`isAdmin`/`isCreator`) and fetches complete peer information only when the peer is incomplete.
- Existing `group info`, member, audit, and mutation commands are unchanged.

### Account logout and login

```text
tg account logout [name] [--yes]
tg account login <name>
```

- Omitting name from logout selects the current account.
- Logout requires interactive confirmation; non-TTY use requires `--yes`.
- Successful logout revokes/resets the Telegram session and marks the account `logged_out` while preserving metadata and `messages.db`.
- A registered account already marked `logged_out` returns idempotent success.
- Terminal authentication/session-expired responses during logout are treated as remotely logged out and allow the local state transition. Network and unexpected errors do not change local state.
- Online commands targeting a logged-out account return `account_logged_out` before client creation.
- Local SQLite commands continue to work for a logged-out current or explicitly selected account.
- Login is only for an existing logged-out account. New identities continue to use `account add`.
- Login authenticates into a temporary session path, then verifies that Telegram `user.id` equals the registered `user_id`.
- Identity mismatch returns `account_identity_mismatch`, destroys the temporary session, and leaves the account logged out.
- Matching authentication atomically replaces the account session and sets `auth_state` to `authenticated`.
- Calling login for an already authenticated account succeeds with `changed: false`.

### Ownership transfer with 2FA

The existing route remains:

```text
tg group admin transfer-owner <chat> <user>
```

- The operation keeps the existing destructive confirmation requirement.
- After confirmation, it requests the current account's 2FA password through hidden TTY input.
- Finite CLI execution uses a reusable secure-input helper that suppresses echo and restores terminal state on success, error, abort, and signal.
- Interactive `listen` uses a masked Ink password prompt after its confirmation dialog.
- Non-TTY execution returns `interaction_required`.
- No password flag or environment-variable fallback exists.
- The service calls mtcute `transferChatOwnership({ chatId, userId, password })`.
- The password is not retained after the call and never appears in output, logs, error details, snapshots, or fixtures.

### Markdown archive

```text
tg archive [chats...]
tg archive --all
  [--output <directory>]
  [--since <relative-or-iso>]
  [--until <relative-or-iso>]
  [--full]
  [--rebuild]
  [--download-media]
```

- At least one chat or `--all` is required.
- Explicit chats and `--all` are mutually exclusive.
- The default output directory is `./telegram-archive`.
- On a first run without a range, archive messages from the previous seven days.
- `--full` removes the lower time bound and conflicts with `--since`.
- `--until` may be combined with either the default window, `--since`, or `--full`.
- For an existing archive, a run without range flags starts after the committed per-chat cursor.
- `--rebuild` recreates the selected chat files. With an existing manifest and no explicit range, it reuses each chat's recorded initial history boundary, including full-history status. Without an existing manifest it uses the normal seven-day default.
- Incremental runs append new messages only. Edits and deletions are reconciled only by `--rebuild`.
- By default, media is represented by type, filename, size, and message ID without downloading bytes.
- `--download-media` downloads available attachments and writes relative links into Markdown.

### Global remote-write access

```text
tg config write-access [status|on|off]
```

- Omitting the action is equivalent to `status`.
- Existing configurations default to enabled.
- `config list` includes the effective `write_access` field.
- When disabled, all Telegram remote mutations fail with `write_access_disabled` before a finite command creates a Telegram client.
- Covered operations include send, edit, Telegram delete, all group mutations, ownership transfer, notification mute/unmute, and folder chat add/remove.
- Authentication, logout, Telegram reads, local SQLite writes, configuration changes, archive file writes, exports, purges, and attachment downloads are not blocked.
- A running `listen` session remains connected for reads, but send/reply and group mutation actions consult the same policy immediately before execution and are rejected while disabled.

## Time Range Semantics

One shared parser handles `read`, `search-online`, and `archive`.

- Relative values use a positive integer plus `s`, `m`, `h`, `d`, or `w`.
- Relative values mean an instant before the command's captured start time. For example, `--since 7d --until 2d` selects from seven days ago through two days ago.
- ISO 8601 values must include `Z` or an explicit UTC offset to avoid host-timezone ambiguity.
- Both bounds are captured once, before network activity, so long-running pagination does not move the window.
- `since` is inclusive and `until` is exclusive.
- `since >= until` returns `invalid_time_range` before client creation.

## Markdown Output

Markdown becomes a fourth finite-command presentation format, not a second structured schema.

The existing `HumanOutput` model remains the common presentation model. A central Markdown renderer covers:

- text
- tables
- detail fields
- summaries with optional tables
- timelines

Message-oriented services produce presentation models containing sender, timestamp, chat, message ID, text, reply context, media-group summary, and attachment metadata. The renderer escapes table pipes, backslashes, and line breaks consistently.

Successful `--markdown` output goes to stdout. Markdown errors are concise human-readable Markdown on stderr and set a non-zero exit status. JSON/YAML failures continue to use stdout and the existing schema-version-1 error envelope.

All explicit format conflicts return `invalid_output_format` before doing work.

## Archive Layout and Consistency

### Layout

Each archive root is bound to one Telegram account identity:

```text
telegram-archive/
  archive-manifest.json
  <marked-chat-id>-<title-slug>.md
  media/
    <marked-chat-id>/
      <message-id>-<safe-filename>
```

The marked chat ID is the stable filename component. The title slug is for readability and may be updated on rebuild without changing manifest identity. Unsafe characters are replaced deterministically, and collisions are disambiguated with the marked ID.

The manifest contains:

```text
schema_version
account_name
account_user_id
created_at
updated_at
chats[marked_chat_id]:
  title
  file
  initial_since
  initial_until
  full_history
  last_message_id
  last_message_date
  last_run
```

Opening an archive with a different Telegram user returns `archive_account_mismatch`. Account aliases may change; `account_user_id` is authoritative.

### Incremental write flow

For each chat independently:

1. Resolve the chat and effective range from flags, manifest, and defaults.
2. Fetch pages with the existing flood-wait normalization and configured page delay conventions.
3. Normalize and chronologically sort messages, media groups, reply metadata, and attachments.
4. Write an invisible stable marker for each Telegram message ID in the Markdown block.
5. Stream the previous file plus deduplicated new blocks into a temporary sibling file.
6. Flush and atomically rename the temporary file over the chat Markdown file.
7. Atomically update the manifest through its own temporary file and rename.

If the Markdown rename succeeds but manifest commit does not, the next run scans message markers while streaming the existing file, recovers the effective maximum archived ID, and avoids duplication. A failed chat leaves its prior file and cursor valid.

Media files use deterministic names. Existing matching files are reused. A media download failure preserves the message Markdown with metadata and records a warning, but the overall command exits as a partial failure because the requested archive is incomplete.

### Partial failure

`archive --all` and multi-chat archive continue after per-chat failures. Successful chat files and cursors remain committed. If any chat or requested attachment fails:

- exit status is non-zero;
- error code is `archive_partial_failure`;
- error details list completed chats, failed chats, files, warnings, and sanitized reasons;
- no password, credential, proxy secret, or session detail appears in the result.

## Account Registry Migration

The registry moves from version 1 to version 2:

```ts
type AccountMeta = {
  name: string
  user_id: number
  username: string
  phone: string
  display_name: string
  auth_state: 'authenticated' | 'logged_out'
}

type AccountRegistry = {
  version: 2
  current_account: string | null
  accounts: AccountMeta[]
}
```

Reading a version-1 registry normalizes each account in memory with `auth_state: 'authenticated'`. All subsequent registry writes use version 2 and the existing atomic/locked write path. Unknown future versions remain errors. `account list` and `account current` expose `auth_state` in all formats.

## Write Access Enforcement

`WriteAccessPolicy` reads the effective configuration and returns a stable failure without printing. Services invoke it at the mutation boundary so interactive and non-interactive callers share enforcement.

Finite command handlers also use a write-command runner that checks the policy before account client construction. The service-level check remains necessary for long-running `listen`, where the setting may change after connection.

The policy is intentionally not a security boundary against a user who can edit local files or run another Telegram client. It is a persistent CLI safety control designed to prevent this CLI and agents using it from performing remote mutations accidentally.

## Error Contract

New stable error codes include:

- `account_logged_out`
- `account_identity_mismatch`
- `interaction_required`
- `write_access_disabled`
- `invalid_time_range`
- `contact_not_found`
- `phone_not_resolvable`
- `folder_not_found`
- `ambiguous_folder`
- `folder_operation_unsupported`
- `invalid_notification_duration`
- `archive_account_mismatch`
- `archive_partial_failure`
- `password_invalid`
- `password_too_fresh`
- `session_too_fresh`

Existing Telegram error normalization remains in use. Ownership transfer maps known mtcute/Telegram RPC errors to the password and session codes above while preserving unknown failures as sanitized `telegram_error` results.

Preflight errors must occur before client creation where possible: invalid limits, time windows, output conflicts, missing archive scope, conflicting archive flags, non-TTY password requirements, confirmations, logged-out accounts, and disabled remote writes.

## Testing Strategy

### Pure unit tests

- Relative and ISO time parsing, captured-now behavior, inclusive/exclusive bounds, and invalid ranges.
- Notification-duration parsing and permanent mute values.
- Folder peer inclusion/exclusion and idempotency.
- Registry v1-to-v2 normalization and version-2 writes.
- Write access defaults and policy decisions.
- Manifest parsing, account binding, cursor selection, filename sanitization, and message-marker deduplication.
- Markdown escaping and rendering for every `HumanOutput` kind.

### Telegram adapter tests

- Mock mtcute calls for dialogs, history, online search, contacts, folders, logout, and ownership transfer.
- Exact typed raw requests for notification get/update calls.
- Incomplete peer handling for administrator-group filtering.
- Phone and username resolution failures.

### Service tests

- Inbox aggregation without read-state mutation.
- Direct reads and searches without SQLite writes.
- Contact list/info normalization.
- Folder ambiguity, shared-folder restrictions, category exclusion, and idempotency.
- Notification status and duration mapping.
- Account logout/login success, network failure, expired session, identity mismatch, and retry.
- Write policy coverage for every remote mutation.

### Archive failure tests

Use temporary directories and injected fakes to cover:

- first seven-day archive;
- full history and explicit ranges;
- normal incremental append;
- rebuild with inherited and overridden ranges;
- interrupted Markdown replacement;
- interrupted manifest replacement;
- retry deduplication after a stale manifest;
- one failed chat among successful chats;
- one failed attachment among successful messages;
- account mismatch;
- deterministic media reuse.

### CLI contract tests

For every new route, cover:

- human success;
- JSON, YAML, and Markdown success;
- output flag conflicts;
- structured and human errors;
- stdout/stderr separation;
- process exit status;
- `--account` selection;
- validation before client construction.

Security regressions assert that write-disabled commands never construct a client, identity mismatch never replaces a session, secure prompts restore terminal state, and passwords never appear in captured output or serialized artifacts.

Final release verification is:

```sh
pnpm test && pnpm typecheck && pnpm build
```

## Implementation Batches

1. Shared foundations: time parser, Markdown output, write policy, registry migration, and adapter interfaces/fakes.
2. Online reads: inbox, read, search-online, contacts, and group list.
3. Telegram settings: notifications, folders, and enforcement across existing writes.
4. Account security: logout/login, hidden input, and 2FA ownership transfer.
5. Archive: manifest, Markdown files, incremental recovery, rebuild/full modes, media, and partial failures.
6. Release completion: English/Chinese README, bundled agent skill reference, changelog, version bump, and full verification.

Each batch lands with focused tests. The npm package is released only after every batch and release verification are complete.

## Acceptance Criteria

- Every command in this design is registered with stable help text and account selection.
- Every finite new and existing command supports explicit Markdown output without changing JSON/YAML contracts.
- Existing commands retain their v0.3 default semantics.
- `read` and `search-online` never write the local message database.
- Inbox does not mark messages read.
- Contact lookup respects Telegram privacy failures.
- Notification and folder mutations obey the global write switch.
- Administrator group discovery includes groups, supergroups, and broadcast channels.
- Logout preserves account metadata and local messages, and login cannot attach a different Telegram identity to them.
- Ownership transfer securely accepts 2FA in TTY and works through mtcute's high-level API.
- Archive requires explicit scope, defaults to seven days, supports full/rebuild/incremental modes, optionally downloads media, survives interrupted metadata commits without duplicates, and returns non-zero on partial failure.
- Registry/config upgrades require no manual migration and preserve v0.3 remote-write behavior.
- Documentation and bundled agent guidance describe online versus local operations, safety controls, archive behavior, and new error codes.
- The complete test, typecheck, and build command passes.
